# SentiVision iOS 기술 의사결정 문서 (개발 착수 전)

## 1) 목적

이 문서는 현재 Python 기반 파이프라인을 iOS 앱으로 옮기기 전에,
필요 기술 스택과 데이터 저장 전략을 확정하기 위한 의사결정 자료다.

기준 문서:
- PRD v1.0의 제품 의도(캔버스 기반 창작 사용자, 분석 API, 피드백 API)를 우선 기준으로 삼는다.

현재 구현 기준:
- 입력 이미지 분석
- saliency 기반 주요 영역 추출
- 주요 색상 추출 (KMeans)
- 감정 예측 (KNN)
- 사용자 피드백 기반 데이터 보강

## 2) 지금 당장 필요한가?

결론: 현재 검증용 프로토타입은 온디바이스만으로도 가능하지만, PRD 기준 MVP는 API 계약을 전제로 설계하는 것이 맞다.

필수:
- iOS UI 프레임워크
- 캔버스 입력 및 팔레트 추출
- 색상 클러스터링
- 감정 분류
- 분석 API / 피드백 API 계약
- 로컬 저장소 또는 서버 저장 전략

선택:
- 클라우드 DB
- 사용자 계정/동기화

권장 해석:
- 구현 순서는 온디바이스 프로토타입 -> FastAPI 연동 MVP로 가져간다.
- 즉, 서버는 "당장 없어도 되는 것"이 아니라 "MVP 제품화 전에 반영할 것"으로 본다.

## 3) 권장 스택 (MVP)

### 3.1 앱 레이어
- UI: SwiftUI
- 비동기: async/await
- 아키텍처: MVVM + UseCase 분리

선정 이유:
- 구현 속도 빠름
- iOS 네이티브 생태계와 궁합이 좋음
- 테스트 단위 분리 용이

### 3.2 이미지 처리
- Vision + Core Image
- 필요 시 Accelerate(vDSP) 보조 사용

역할:
- 캔버스 픽셀/팔레트 처리
- 색공간/픽셀 데이터 처리
- saliency 유사 정보 추출

### 3.3 색상 추출/클러스터링
- 1차: Swift 구현 KMeans (k=3)
- 고도화: Accelerate 최적화

### 3.4 감정 분류
- 1차: Swift 거리기반 KNN 구현 (작은 데이터셋에 적합)
- 2차: Core ML 모델 전환 (성능/유지보수 향상)

## 4) DB 선택

### 4.1 추천 우선순위
1. SwiftData (iOS 17+ 타깃)
2. Core Data (하위 iOS 버전 지원 필요 시)
3. 파일 기반(JSON/CSV) (프로토타입 한정)

현재 프로젝트 상황에서는 SwiftData가 가장 현실적.

### 4.2 저장 대상 데이터
- 분석 로그
  - 이미지 식별자
  - 대표 RGB 3개
  - 예측 감정
  - 사용자 수정 감정
  - 생성 시각
- 학습 샘플 확장 데이터
  - R, G, B
  - emotion
  - source (system/user)

## 5) 서버/클라우드가 필요한 시점

PRD 기준 MVP부터 아래 항목은 API 계층으로 설계한다:
- 분석 요청 (`POST /analyze`)
- 피드백 저장 (`POST /feedback`)
- 상태 확인 (`GET /health`)

아래 요구가 생기면 클라우드/영속 인프라를 확장한다:
- 여러 기기 간 동기화
- 사용자별 데이터 백업
- 실시간 모델/라벨 업데이트
- 운영 대시보드/통계

도입 후보:
- 빠른 시작: Firebase 또는 Supabase
- 커스텀 운영: FastAPI + PostgreSQL + Object Storage

## 6) 기술 선택 매트릭스

### 6.1 로컬 저장
- SwiftData
  - 장점: 최신 iOS 친화적, 코드 간결
  - 단점: 타깃 버전 제약
- Core Data
  - 장점: 안정성, 하위 버전 폭넓은 지원
  - 단점: 초기 보일러플레이트 많음

권장 결정:
- 타깃이 iOS 17 이상이면 SwiftData
- 그 미만 포함이면 Core Data

### 6.2 분류 로직
- Swift KNN 직접 구현
  - 장점: 빠른 MVP, 디버깅 쉬움
  - 단점: 데이터 커지면 성능 저하
- Core ML 변환
  - 장점: 추론 성능/배포 품질 우수
  - 단점: 모델 변환/검증 공정 추가

권장 결정:
- 1차 릴리스: Swift KNN
- 2차 릴리스: Core ML 전환

## 10) 사용자 결정사항 반영 (2026-03-18)

아래 항목은 현재 기준으로 확정한다.

1. 감정은 PRD의 결과 시각화 구조에 맞춰 예측 감정 1개와 상위 점수 분포로 제시한다.
2. 사용자는 필요 시 실제 감정과 메모를 입력해 피드백을 제출할 수 있다.
3. 사용자 피드백은 검증 후 데이터셋에 반영한다.
4. 분석 트리거는 기본적으로 사용자의 `분석하기` 액션으로 시작한다.
5. 초기에는 API 연동 가능한 MVP 구조로 개발하고, 추후 전시/공유 기능을 추가한다.

## 11) 감정 체계 (초기 운영 기준)

### 11.1 상위 그룹
- 긍정(Positive)
- 부정(Negative)
- 중립(Neutral)

### 11.2 하위 감정 라벨 예시
- 긍정: 기쁨, 편안함, 설렘, 희망
- 부정: 불안, 분노, 슬픔, 답답함
- 중립: 차분함, 무감정, 집중, 관찰

운영 원칙:
- 초기 릴리스에서는 대표 감정 집합을 운영하되, PRD의 피드백 루프에 따라 확장 가능성을 열어둔다.
- 라벨 변경이 필요하면 버전 단위(예: v1 -> v2)로 마이그레이션한다.

## 12) 피드백 검증 후 반영 정책

### 12.1 왜 검증이 필요한가
- 사용자가 즉흥적으로 선택한 라벨이 바로 반영되면 데이터셋 오염이 발생할 수 있다.
- 오염된 샘플은 모델 품질을 빠르게 저하시킨다.

### 12.2 반영 파이프라인
1. 사용자 피드백 저장
2. 검증 점수 계산
3. 반영 후보 큐에 적재
4. 임계치 충족 시 데이터셋 반영
5. 반영 이력 로그 저장

### 12.3 검증 점수 규칙 (초안)
- 점수 범위: 0~100
- 구성:
  - 일치도 점수(최대 40): 사용자가 "결과가 맞다"에 가깝게 응답할수록 가산
  - 신뢰도 점수(최대 20): 동일 사용자의 최근 응답 변동성이 낮으면 가산
  - 반복성 점수(최대 25): 유사 색상 군집에서 동일 라벨이 반복되면 가산
  - 데이터 품질 점수(최대 15): 입력 이미지 품질(너무 작은 영역/노이즈 여부) 반영

반영 기준:
- 점수 70 이상: 반영 후보
- 점수 85 이상: 즉시 반영 가능
- 점수 70 미만: 데이터셋 미반영, 로그만 저장

### 12.4 추가 안전장치
- 동일 세션 연속 입력은 최대 반영 수 제한 (예: 1일 20건)
- 극단적 편향 라벨 비율 감지 시 자동 보류
- 원본 데이터셋과 사용자 반영 데이터셋 분리 저장

## 13) 자동 분석 트리거 정책

기본 동작:
- 사용자가 `분석하기`를 눌렀을 때 분석 실행

예외 처리:
- 유효 팔레트가 부족하면 분석 요청을 막고 안내 메시지 제공
- 자동 분석은 추후 실험 기능으로 검토 가능

사용자 경험:
- 단계형 로딩 메시지(색상 추출 중 -> 감정 점수 계산 중) 표시
- 필요 시 "다시 분석" 버튼 제공

## 14) API 우선 MVP, 이후 공유 기능 확장 전략

### 14.1 1단계 (MVP)
- 앱은 캔버스 입력과 결과/피드백 UI를 제공
- 분석 및 피드백은 FastAPI 또는 동등한 API 계층과 연동
- 로컬 DB는 히스토리/임시 저장 용도로 병행 사용 가능

### 14.2 2단계 (공유 기능)
- 전시/미술관 형태 공유를 위한 기능 추가:
  - 작품 업로드
  - 감정 태그 노출
  - 공개/비공개 설정
  - 큐레이션 피드

### 14.3 3단계 (서버 도입)
- 동기화 및 공유 고도화를 위한 백엔드 확장
- 권장 구성:
  - API: FastAPI 또는 BaaS(Firebase/Supabase)
  - DB: PostgreSQL
  - 이미지 저장: Object Storage
