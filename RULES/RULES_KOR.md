# SentiVision iOS 기술 의사결정 문서 (개발 착수 전)

## 1) 목적

이 문서는 현재 Python 기반 파이프라인을 iOS 앱으로 옮기기 전에,
필요 기술 스택과 데이터 저장 전략을 확정하기 위한 의사결정 자료다.

현재 구현 기준:
- 입력 이미지 분석
- saliency 기반 주요 영역 추출
- 주요 색상 추출 (KMeans)
- 감정 예측 (KNN)
- 사용자 피드백 기반 데이터 보강

## 2) 지금 당장 필요한가?

결론: MVP 단계에서는 서버/클라우드 없이 온디바이스만으로 시작 가능.

필수:
- iOS UI 프레임워크
- 이미지 처리
- 색상 클러스터링
- 감정 분류
- 로컬 저장소

선택:
- 백엔드 API
- 클라우드 DB
- 사용자 계정/동기화

## 3) 권장 스택 (MVP)

### 3.1 앱 레이어
- UI: SwiftUI
- 비동기: async/await
- 아키텍처: MVVM + UseCase 분리

선정 이유:
- 구현 속도 빠름
- iOS 네이티브 생태계와 궁합이 좋음
- 테스트 단위 분리 용이

### 3.2 이미지 처리
- Vision + Core Image
- 필요 시 Accelerate(vDSP) 보조 사용

역할:
- 이미지 리사이즈
- 색공간/픽셀 데이터 처리
- saliency 유사 정보 추출

### 3.3 색상 추출/클러스터링
- 1차: Swift 구현 KMeans (k=3)
- 고도화: Accelerate 최적화

### 3.4 감정 분류
- 1차: Swift 거리기반 KNN 구현 (작은 데이터셋에 적합)
- 2차: Core ML 모델 전환 (성능/유지보수 향상)

## 4) DB 선택

### 4.1 추천 우선순위
1. SwiftData (iOS 17+ 타깃)
2. Core Data (하위 iOS 버전 지원 필요 시)
3. 파일 기반(JSON/CSV) (프로토타입 한정)

현재 프로젝트 상황에서는 SwiftData가 가장 현실적.

### 4.2 저장 대상 데이터
- 분석 로그
  - 이미지 식별자
  - 대표 RGB 3개
  - 예측 감정
  - 사용자 수정 감정
  - 생성 시각
- 학습 샘플 확장 데이터
  - R, G, B
  - emotion
  - source (system/user)

## 5) 서버/클라우드가 필요한 시점

다음 요구가 생기면 도입:
- 여러 기기 간 동기화
- 사용자별 데이터 백업
- 실시간 모델/라벨 업데이트
- 운영 대시보드/통계

도입 후보:
- 빠른 시작: Firebase 또는 Supabase
- 커스텀 운영: FastAPI + PostgreSQL + Object Storage

## 6) 기술 선택 매트릭스

### 6.1 로컬 저장
- SwiftData
  - 장점: 최신 iOS 친화적, 코드 간결
  - 단점: 타깃 버전 제약
- Core Data
  - 장점: 안정성, 하위 버전 폭넓은 지원
  - 단점: 초기 보일러플레이트 많음

권장 결정:
- 타깃이 iOS 17 이상이면 SwiftData
- 그 미만 포함이면 Core Data

### 6.2 분류 로직
- Swift KNN 직접 구현
  - 장점: 빠른 MVP, 디버깅 쉬움
  - 단점: 데이터 커지면 성능 저하
- Core ML 변환
  - 장점: 추론 성능/배포 품질 우수
  - 단점: 모델 변환/검증 공정 추가

권장 결정:
- 1차 릴리스: Swift KNN
- 2차 릴리스: Core ML 전환

## 10) 사용자 결정사항 반영 (2026-03-18)

아래 항목은 현재 기준으로 확정한다.

1. 감정 카테고리는 고정한다.
2. 감정은 목록에서 사용자가 선택할 수 있게 하며, 상위 그룹(긍정/부정/중립 등)으로 나누어 제공한다.
3. 사용자 피드백은 검증 후 데이터셋에 반영한다.
4. 분석 트리거는 자동 분석으로 시작한다.
5. 초기에는 오프라인 앱으로 개발하고, 추후 전시/공유 기능을 추가한다.

## 11) 감정 체계 (고정 라벨)

### 11.1 상위 그룹
- 긍정(Positive)
- 부정(Negative)
- 중립(Neutral)

### 11.2 하위 감정 라벨 예시
- 긍정: 기쁨, 편안함, 설렘, 희망
- 부정: 불안, 분노, 슬픔, 답답함
- 중립: 차분함, 무감정, 집중, 관찰

운영 원칙:
- 앱 릴리스 중에는 라벨 집합을 변경하지 않는다.
- 라벨 변경이 필요하면 버전 단위(예: v1 -> v2)로 마이그레이션한다.

## 12) 피드백 검증 후 반영 정책

### 12.1 왜 검증이 필요한가
- 사용자가 즉흥적으로 선택한 라벨이 바로 반영되면 데이터셋 오염이 발생할 수 있다.
- 오염된 샘플은 모델 품질을 빠르게 저하시킨다.

### 12.2 반영 파이프라인
1. 사용자 피드백 저장
2. 검증 점수 계산
3. 반영 후보 큐에 적재
4. 임계치 충족 시 데이터셋 반영
5. 반영 이력 로그 저장

### 12.3 검증 점수 규칙 (초안)
- 점수 범위: 0~100
- 구성:
  - 일치도 점수(최대 40): 사용자가 "결과가 맞다"에 가깝게 응답할수록 가산
  - 신뢰도 점수(최대 20): 동일 사용자의 최근 응답 변동성이 낮으면 가산
  - 반복성 점수(최대 25): 유사 색상 군집에서 동일 라벨이 반복되면 가산
  - 데이터 품질 점수(최대 15): 입력 이미지 품질(너무 작은 영역/노이즈 여부) 반영

반영 기준:
- 점수 70 이상: 반영 후보
- 점수 85 이상: 즉시 반영 가능
- 점수 70 미만: 데이터셋 미반영, 로그만 저장

### 12.4 추가 안전장치
- 동일 세션 연속 입력은 최대 반영 수 제한 (예: 1일 20건)
- 극단적 편향 라벨 비율 감지 시 자동 보류
- 원본 데이터셋과 사용자 반영 데이터셋 분리 저장

## 13) 자동 분석 트리거 정책

기본 동작:
- 사용자가 드로잉을 멈춘 뒤 짧은 대기 시간(예: 0.8초) 후 자동 분석 실행

예외 처리:
- 캔버스 변화량이 너무 작으면 분석 생략
- 연속 드로잉 중에는 분석 호출을 디바운스하여 과호출 방지

사용자 경험:
- 자동 분석 상태 인디케이터(분석 중/완료) 표시
- 필요 시 "다시 분석" 수동 버튼 제공
